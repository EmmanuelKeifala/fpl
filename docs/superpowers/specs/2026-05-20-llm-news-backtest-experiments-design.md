# LLM News Backtest Experiments Design

## Goal

Evaluate whether a hybrid LLM-assisted FPL manager can improve season replay performance across `2021-2022` through `2024-2025`, and measure how much automated historical news context changes decisions.

## Scope

V1 is a small smoke matrix, not a large optimization search. It proves the experiment plumbing, produces comparable reports, and keeps cost controlled before scaling.

The first matrix covers:

- Seasons: `2021-2022`, `2022-2023`, `2023-2024`, `2024-2025`.
- Comparator: existing deterministic `fair` strategy.
- Experimental modes: `llm-news-strict` and `llm-news-loose`.
- Config count: 3-5 prompt/model/parameter combinations.

## Strategy Model

Use a hybrid candidate-ranker design. Deterministic code remains responsible for FPL legality and candidate generation. The LLM does not invent arbitrary squads from scratch in V1.

For each gameweek, the engine will provide the LLM with:

- Current manager state: squad, bank, free transfers, available chips, previous decisions.
- Point-in-time player/fixture snapshot.
- A bounded list of legal candidate decisions generated by deterministic code.
- News context for the current gameweek and selected mode.

The LLM returns a ranked candidate choice plus a short explanation. The replay applies only the selected prevalidated candidate decision.

## News Modes

`llm-news-strict` only uses historical items whose publish timestamp is known and before the gameweek deadline. This mode is intended to remain point-in-time fair.

`llm-news-loose` may use broader around-gameweek public context. It must be clearly labeled as not strictly fair because timestamps and availability may be imperfect.

Both modes must record news provenance in the experiment report: source URL, title, publisher, published timestamp when available, retrieval timestamp, and any fairness warnings.

## Automated News Ingestion

Automated ingestion should start as best-effort and cache-first. The system may query public web/news sources for player injuries, suspensions, lineup hints, manager comments, and fixture-related context.

Fetched news must be cached locally by season/gameweek/mode so reruns do not repeatedly hit external services and so experiment reports are reproducible. If news fetching fails, the experiment should continue with an explicit missing-news warning rather than failing the whole season replay.

## Experiment Runner

Add a runner that can prepare missing season caches, run each configured season/mode/config combination, and write a consolidated report.

The report should include:

- Total points by season and config.
- Average total points across seasons.
- Per-season deltas against deterministic `fair`.
- Transfers, chips, captain points, bench points.
- News mode and fairness label.
- LLM model/prompt/config metadata.
- Warnings for missing data, missing news, or loose-mode leakage risk.

## Cost And Safety

V1 must default to a small matrix. It should require explicit CLI flags for LLM/news execution so normal deterministic backtests do not accidentally incur model or web costs.

LLM calls should support caching keyed by prompt input hash, model, and config. Cached responses should be reused unless the user explicitly disables cache.

## Success Criteria

- The deterministic `fair` comparator can run for all four seasons.
- `llm-news-strict` and `llm-news-loose` can run at least one smoke config across all four seasons.
- Reports make strict versus loose fairness obvious.
- All applied decisions are legal through the existing replay validator.
- Failures in news retrieval or LLM calls are surfaced in report warnings with enough detail to diagnose.

## Non-Goals For V1

- Full automated large-scale hyperparameter search.
- Direct LLM-created arbitrary transfer plans.
- Guaranteed complete historical news coverage.
- Claiming strict fairness for loose news experiments.
